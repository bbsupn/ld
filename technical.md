# 技术概览

[← 返回首页](./)

本页面介绍项目的技术实现，面向对逆向工程和游戏服务端复现感兴趣的读者。

---

## 基本原理

QQ雷电的客户端在停服后仍能运行，只是连不上已关闭的官方服务器。客户端并不关心服务器"是谁"，它只认协议——只要有程序能按它期望的格式回话，它就照常工作。

因此我们的核心工作不是改游戏，而是**让客户端误以为官方服务器还在**：

1. **逆向**客户端与服务器之间的私有协议——消息分帧、加密、各功能 opcode 的语义
2. **实现**一套兼容服务端，对客户端的每一类请求给出格式正确的应答
3. **桥接**客户端与服务端——客户端只会连本地命名管道，由启动器把管道流量转译并转发到远程服务端

全程不改动客户端磁盘文件，只在运行时按需做内存修补。客户端始终以为自己在和官方服务器对话。

## 系统架构

项目由三个独立组件构成，跨越 32 位与 64 位、本地与远程两条边界：

```
   玩家电脑                                          远程服务器
┌──────────────────────────────────┐          ┌──────────────────┐
│  启动器 (.NET, x64)                │          │  服务端 (.NET)     │
│  ├─ 大厅 GUI（登录/选房/准备）       │◄── gRPC ─►│  ├─ 大厅逻辑       │
│  └─ 管道桥 PipeBridgeServer        │  (TLS)    │  ├─ 座位租约       │
│      └─ DefaultPacketHandler      │          │  └─ SQLite 数据   │
│          (协议处理在启动器进程内)     │          └──────────────────┘
└──────┬──────────────────▲────────┘
       │ 拉起+注入          │ 命名管道 (TEA 加密帧)
       ▼                  │
┌──────────────────────────────────┐
│  游戏客户端 Client.exe (原版, 32 位) │
│  └─ 注入 ClientMods.dll (运行时修补) │
└──────────────────────────────────┘
```

关键设计：**协议处理（`DefaultPacketHandler`）跑在启动器进程内，而不是远程服务端**。客户端的命名管道字节流到启动器为止，启动器把需要持久化或多人同步的部分通过 gRPC 转发给远程服务端。远程服务端只负责大厅状态、座位租约和玩家数据，从不直接接触管道字节流。这条边界让协议适配层贴近客户端，服务端保持纯粹的数据与状态权威。

## 协议逆向

客户端与服务器之间是一套私有二进制协议，全部从零逆向还原。

**分帧与加密**
- 客户端通过命名管道收发消息，外层按 `MessageType` 分两类：`0`（心跳/广播风格）与 `2`（会话数据风格）
- 消息载荷用 **TEA 分组加密**，密钥固定内置于客户端；我们还原了密钥与加密例程，可对任意方向的帧解密、改写、重新加密
- 解密后的载荷以 `opcode`（或 `kind`）首字节区分功能，进入客户端各自的处理函数

**握手与初始化序列**
- 连接建立后客户端发 HELLO，服务端必须按固定顺序回 `HELLO ACK → kind=0x02（建立会话）→ kind=0x01（状态写入）→ kind=0x11（实体状态）`，客户端才会渲染出完整的游戏桌台 UI 与"准备游戏"按钮
- 顺序、字段、长度任何一处不对，客户端要么静默丢弃要么崩溃——例如早期发现：对人类玩家 UIN 发 `kind=0x01` 会触发客户端把整数 UIN 当对象指针解引用，导致除零崩溃（`0xC0000094`）

**已还原的功能 opcode（部分）**
- 角色选择、装备同步（0x14/0x15/0x16）、实时换装（0x97）
- 仓库装备与物品两个分页、商城列表下发（0x28）、购买请求与应答（0x29/kind126）
- 成长结算：经验（0x24，写入玩家对象 +0x230）、军衔积分（kind0x02，写入 +0x38）
- 准备/开始游戏的状态机、心跳保活（opcode 0xA4，约 19 秒一次）

**资源包**
- 游戏资源封装在 7 个加密资源包中，我们破解了 XOR 加密与索引格式，提取出全部 **6,527 个文件**
- 物品主表通过解析 `ItemManager.ib` 完整还原，得到 **519 个唯一物品定义**（一对一映射，含装备与各类道具），按 ID 区间区分武器/机体/僚机/保险/道具/强化材料/合成材料

这些分析是服务端的根基——只有完全理解客户端"说的话"，服务端才能正确"回话"。

## 运行时修补

有些客户端行为无法靠协议绕过——例如角色选择的回包逻辑、战斗场景的初始化、键盘输入路径。这些只能在客户端进程内直接干预。我们的做法是注入一个独立的 C++ DLL（`ClientMods`），在运行时对 Client.exe 打补丁，而**不改动它的磁盘文件**。

**修补手段**
- **字节 patch**：直接覆写指令字节（如跳过某次校验、改写立即数）
- **call detour**：把原始 call 改指向我们的函数，处理后再跳回原逻辑
- **naked shim**：用裸函数手写汇编桥接，保护好寄存器与栈平衡后用 `push/ret` 跳转，避免破坏调用约定

目前 DLL 维护着 **90+ 个补丁/detour 锚点**，覆盖稳定性修复、角色选择、输入、战斗场景加载、商城购买等。

**工程约束**
- DLL **不依赖 CRT**——静态 CRT 在某些宿主下初始化会失败，因此自定义入口、禁用默认库、用 `__movsb` 代替 `memcpy`、不用全局构造器
- 注入前对 Client.exe 做**签名校验**：DLL 内置一组关键地址的预期字节（由构建脚本从参考客户端提取），运行时逐一比对，版本不符立即中止，避免把补丁打到错误的地址上
- 启动器侧反过来对 DLL 做 **SHA256 指纹校验**，确保注入的是构建产物本身

**跨位宽的坑**
- 启动器是 64 位进程，Client.exe 是 32 位。跨进程枚举模块必须用 `EnumProcessModulesEx(..., LIST_MODULES_ALL)`；启动器自己的 64 位 `kernel32` 地址对目标进程无效，需从 `SysWOW64` 下的 32 位 `kernel32.dll` 的 PE 导出表重新计算 RVA，再加上目标进程里的模块基址

## 数据权威与持久化

玩家数据（角色、装备、仓库、经验、军衔积分）的**唯一权威是远程服务端**，存于 SQLite。

- 启动器进程内的协议处理层通过抽象接口（如 `IEquipmentStore`、`ICharacterStore`）访问数据，实现走 gRPC 调远程服务端，启动器本地不再保存这些状态
- 这让"协议适配"与"数据权威"彻底解耦：换装、购买、成长结算都在服务端落库，玩家换台机器或重登都不丢
- 商城购买是个典型的跨进程闭环：购买请求到达启动器进程内的协议层 → 经 gRPC 转发到服务端原子扣减军衔积分并发货入库 → 应答回客户端解除等待、弹出获得提示

## 启动器的两条路线

启动器目前有两套实现，并行演进：

**稳定版（WinForms / C#）** — 当前主力，功能完整、经过端到端验证，承担登录、选房、座位管理、拉起客户端、管道桥接与运行时注入的全部职责，与服务端走 gRPC。

**大厅版（WinUI 3 / C++，实验性）** — 下一代实现，目标是把"启动器"和"游戏大厅"融合：不只是登录后选房的工具，而是在启动器内还原接近游戏内大厅的体验。它接入了原版的桌台资源——读取客户端目录下的桌台布局配置（`table.ini`）、桌台底图、叠加图与悬停图，按配置里的桌台/座位矩形做命中区，复现"选房间 → 看桌台 → 点空座入座"的原大厅式交互；试炼场的 AI 固定占左座，对局中或满桌的桌台不可点，进行中的桌台可展示地图预览。

需要说明的是，这里复用的是原版的**界面资源与交互模型**，而非原版网络协议——选座后的座位租约、保活、准备、开局走的是项目自己的服务端接口。

协议上大厅版也做了演进——与服务端之间不走 gRPC，而是一套自定义的 **TCP/TLS 帧协议**（`QLD1` 二进制信封 + SHA256 证书固定，载荷暂用 JSON）。过渡期内服务端同时提供 gRPC 与新协议两个端点。目前大厅版已实现登录注册、玩家信息、按区服/星级拉取房间、桌台选择与入座、座位租约（加入/保活/离开/准备）、客户端拉起与管道桥接、ClientMods 注入，桌台视图仍在持续完善。它不替换、也不会回退稳定版——稳定版始终保持可用，大厅版只在功能验证后逐步接管。

## 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 服务端 | C# / .NET 10 | 大厅逻辑、座位租约、数据权威 |
| 启动器（稳定版） | C# / .NET 10（WinForms） | 当前主力，当前发布 x64，后续计划提供 x86 |
| 启动器（大厅版） | C++ / WinUI 3（WinRT） | 实验性，启动器与大厅融合，走自定义 TCP/TLS |
| 运行时补丁 | C++（x86 DLL） | 注入 32 位客户端，免 CRT |
| 数据库 | SQLite | 轻量级，无需额外安装 |
| 通信 | gRPC (TLS) + 自定义 TCP/TLS + 命名管道 | 启动器↔服务端走 gRPC 或 QLD1 帧，客户端↔启动器走本地管道 |

> 协议处理层（`Server.Core`）由启动器进程内承载，与服务端共享同一套代码库，但运行在玩家本地。
