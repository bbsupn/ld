## 基本原理

QQ雷电的客户端（游戏本体）在停服后仍然可以运行，只是无法连接到已关闭的官方服务器。

我们的做法是：
1. **分析**原版客户端与服务器之间的通信协议
2. **编写**一套全新的服务端程序，能够正确响应客户端的请求
3. **提供**一个启动器，帮助玩家方便地连接到我们的服务端

整个过程不修改游戏客户端本体，客户端以为自己在和"官方服务器"通信。

## 系统架构

```
  玩家电脑                          服务器
┌──────────┐                  ┌──────────┐
│  启动器   │ ◄──── 网络 ────► │  服务端   │
│ (大厅UI)  │                  │ (联机逻辑) │
└────┬─────┘                  └────┬─────┘
     │ 启动                        │
     ▼                             │
┌──────────┐                       │
│ 游戏客户端 │ ◄──── 本地通道 ────────┘
│ (原版QQ雷电)│
└──────────┘
```

- **启动器**在玩家电脑上运行，提供登录、选房间等大厅功能
- **服务端**可以部署在任意服务器上，处理联机逻辑和数据存储
- **游戏客户端**就是原版程序，通过本地通道与服务端通信

玩家只需要下载启动器，服务端由社区统一部署维护。

---

## 逆向工程

项目的核心工作是对原版客户端进行逆向分析：

- **通信协议**：完整还原了客户端与服务器之间的消息格式和加密方式，包括登录、心跳、角色选择、准备游戏等全部流程
- **游戏资源**：破解了资源包的加密和压缩方案，提取出全部 6,527 个游戏文件（配置、模型、贴图、UI 等）
- **数据格式**：解析了装备定义、关卡配置、敌机属性等核心数据结构

这些分析成果是实现服务端的基础——只有完全理解客户端"说的话"，服务端才能正确"回答"。

## 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 服务端 | C# / .NET 8 | 跨平台，可部署在 Windows 或 Linux |
| 启动器 | WinForms / .NET 10 | Windows 桌面应用 |
| 数据库 | SQLite | 轻量级，无需额外安装 |
| 通信 | gRPC + 本地管道 | 启动器走网络，客户端走本地 |
